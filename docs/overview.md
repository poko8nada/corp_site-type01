# コーポレートサイト型フォーマット（type01）

## Problem

中小企業向けのコーポレートサイトでは、利用者が欲しい情報にすぐ辿り着けることが重要だが、案件ごとにゼロから作るとコストと品質のばらつきが大きくなりやすい。本プロジェクトは、納品しやすい静的成果物と、パターンを差し替えられる再利用可能な型を用意し、制作側とクライアント双方の負担を下げることを目的とする。

## Goals

- 一般的な中小企業サイトの構成（お問い合わせを含む）をカバーできるテンプレートとして拡張できること
- ビルド成果物を、GitHub Pages や国内で一般的なレンタルサーバーなど静的ホスティングに載せるだけで運用できる形に寄せること
- 社名・住所・トップメッセージ・サイトカラー等を構造化データ（JSON 等）から注入し、案件ごとに差し替えやすくすること
- GTM（GA4 利用想定）の置き場、sitemap など、コーポレートサイトに必要になりがちな基盤要素を揃えること
- Cloudflare 上にデプロイし、非公開に近い形でステージングや最終確認ができる導線を確保すること（運用設計のゴール）
- デザインを Pencil（`.pen`）で管理し、実装と同期しやすくすること

## Non-goals

- ランディングページ（LP）特化の演出や構成（一般的なコーポレートサイトの範囲に留める）
- 派手なアニメーションを主目的とした表現（ブランディングは重視しつつ、情報アクセスの速さと分かりやすさを優先する）
- 本リポジトリ内での CMS 直結（ブラウザから CMS API を直接叩く方式や、レンタルサーバー上の PHP プロキシ等の実装は対象外。必要なら別システム・別リポジトリで扱う想定）
- お知らせ・記事の高頻度更新を前提とした CMS 内蔵（静的納品を主軸とし、更新が主戦場になる場合は別途 CMS や BFF を想定）

## Constraints

- 納品・本番の主軸は静的配信（GitHub Pages 等）。サーバーサイドの常時実行は、ホスティング先の制約に依存する
- Cloudflare は主に確認用環境として使う想定であり、顧客との非公開レビューができる運用（アクセス制御の具体手段はインフラ側の選択に依存）を満たす必要がある
- 将来 CMS を繋ぐ場合でも、API キーのフロント露出、CORS、レイテンシは設計上避ける（方針は `docs/draft.md` の議論メモに準拠し、本リポジトリでは未実装）

## Tech Stack Rationale

- Honox と Hono：ファイルベースのルーティングとアイランド（部分的なクライアント活性化）により、コーポレートサイト向けの段階的なインタラクティブ化と開発体験の両立を図る
- Vite と `@hono/vite-ssg`：静的サイト生成で配信物を静的ファイル中心に寄せ、GitHub Pages や静的ホスティングとの親和性を高める
- Cloudflare Workers（Wrangler）：`wrangler deploy` によるエッジ配信とプレビュー環境の標準化。`compatibility_date` と `nodejs_compat` で実行環境の互換性を明示する
- Tailwind CSS v4（`@tailwindcss/vite`）：デザイントークンとユーティリティで、コンポーネントパターンを増やしても見た目の一貫性を保ちやすくする
- TypeScript と oxfmt / oxlint、Vitest：型安全と軽量なツールチェインで、テンプレートの複製や改修時の保守コストを抑える

主要な技術選定の比較や却下案の詳細は、必要に応じて `docs/adr/` に ADR として追加する（現時点で ADR 番号へのリンクは置かない）。
